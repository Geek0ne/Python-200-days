# Day 026 — 学习笔记

## 正则表达式的心法

正则不是死记硬背的，它的本质是**有限状态自动机**。每次匹配就是走一条路径，回溯就是走岔路回头重试。

**三个核心思维转变：**

1. **从"匹配具体字符"到"匹配模式"** — 不再问"是不是'Hello'"，而是问"是不是大写字母开头、后跟小写字母的单词"
2. **从"遍历"到"声明"** — 不再写for循环，而是声明"我要什么模式"
3. **从"精确"到"容错+精确"** — 正则的核心价值在于处理"不确定但可描述"的文本

## 灾难性回溯 — 最隐蔽的性能杀手

```
危险信号: (a+)+b, (.*)*, (.+)+
特点: 内层量词 + 外层量词 + 固定结尾不匹配
结果: 匹配失败时 O(2^n) 回溯
```

**经验法则**：如果一个正则用了超过两层量词嵌套，它一定有更简单的写法。

## 编码哲学

Python 3 的 str/bytes 分离是革命性的。它强制你思考：

- "这个数据是文本还是字节？"
- "它从哪里来？用的是什么编码？"
- "我要往哪里去？目标编码是什么？"

**大多数编码错误不是技术问题，而是"不思考"的问题。**

## 日志解析的架构启发

实战中的 LogFilter 链式调用展示了**管道模式**（Pipeline Pattern）的优点：

```text
原始数据 → 解析 → 过滤 → 转换 → 输出
```

每一层只做一件事，可以任意组合。这种模式在 ETL、数据处理管道中非常常见。

## 常见坑点汇总

| 坑 | 症状 | 根因 |
|----|------|------|
| 忘记 r 前缀 | `\b` 变成退格符 | Python 先解释反斜杠 |
| 贪婪匹配 | 匹配范围过大 | 默认贪婪，需要加 `?` |
| 中文匹配失败 | `\w` 不匹配中文 | `\w` 默认= `[a-zA-Z0-9_]` |
| 编码错误 | `UnicodeDecodeError` | 文件实际编码 ≠ 假设编码 |
| `re.match` vs `re.search` | 明明有却匹配不到 | match 从开头匹配 |
| 灾难性回溯 | 正则卡死 | 嵌套量词导致指数级回溯 |
